### V4.2 冒烟测试 · 2026-06-21 21:00 · 元一

#### 测试环境
- Playwright + Chrome for Testing (headless)，两个隔离 context 模拟 A/B 双端
- 串行启动 A、B 两个浏览器实例（避免资源冲突）
- 截图：/tmp/smoke-1-B-initial.png（5 张手牌正常）

#### 验证结果
- ✅ **Bug 1（手牌丢失）**：B 端初始手牌 = 5 张，完全修复
- ✅ **回合切换**：A 结束回合后 turn 正确切到 'ai'
- ✅ **状态同步**：A 能看到 B 部署
- ❌ **新 Bug 4（K 值不补）**：B 端在 A 结束回合后 `state.k.ai = {cur:0, max:0}`

#### Bug 4 根因（源码核对）
1. A 端 `newGame('online')` 时设 `state._startTurnDone = true`（line 1040）
2. A 端 `endTurn()` 联机分支**不调 startTurn**，也没重置 `_startTurnDone`（line 1103-1107）
3. 第二次 `onlineSendState` 时 `_startTurnDone` 仍是 `true`
4. B 端 `onReceiveState` 看到 `state._startTurnDone === true`，走"已由对手执行"分支（line 970-973），**跳过自己的 startTurn**
5. B 端 K 永远不补，无法部署/攻击

#### 修复建议（待知总拍板）
- **方案 A**（推荐，最自然）：A 端 `endTurn` 联机模式也调 `startTurn(next)`，跟人机/PVP 一致；B 端只需 render
- **方案 B**（最小改动）：A 端 `endTurn` 后将 `state._startTurnDone = false`，让 B 端 `onReceiveState` 走 else 分支调 startTurn
- 顺便：startTurn 内部也应该设 `state._startTurnDone = true`（这样在 A 端调 startTurn 后 B 端不会重复）

- `/tmp/smoke-1-B-initial.png`：B 端第 1 回合初始 5 张手牌（伯克级/战斧/弗吉尼亚/海狼/福特级）✅

## 七、联机对战技术说明（供开发者参考）

### WebSocket 通信协议

**客户端 → 服务端**

| 消息类型 | 说明 |
|---------|------|
| `create_room` | 创建房间，服务端返回 `room_created` |
| `join_room` | 加入房间（带 `roomId`），成功后双方收到 `game_start` |
| `rejoin` | 断线重连（带 `roomId` 和 `side`），60 秒内有效 |
| `state` | 发送游戏状态给对手（己方手牌已隐藏） |
| `ping` | 客户端心跳，服务端回应 `pong` |

**服务端 → 客户端**

| 消息类型 | 说明 |
|---------|------|
| `room_created` | 房间创建成功，返回 `roomId` |
| `game_start` | 两人到齐，通知双方开始（返回 `youAre` 阵营） |
| `state` | 对手发来的游戏状态 |
| `opponent_left` | 对手断开（客户端启动 60 秒判胜倒计时） |
| `rejoin_ok` | 重连成功 |
| `error` | 错误信息 |
| `pong` | 心跳回应 |

### 状态同步逻辑

- 房主（`player` 阵营）创建房间并初始化游戏状态
- 房主调用 `startTurn()` 后，通过 `onlineSendState()` 将状态发给加入者
- 加入者收到状态后，若轮到自己则调用 `startTurn()`，再发回更新后的状态
- 手牌隐藏：发送时己方手牌替换为 `__hidden__`，接收时用本地保存的真实手牌恢复
