## 〇、背景与目标

### 0.1 V4.x 的根本问题

V4.x 沿用 V2 的人机对战代码架构，把"联机"当成"人机的特例"：

```js
// V4.x 的核心问题
state.turn = 'player' | 'ai'           // 永远只有这两种值
state.player = 房主（A 端）
state.ai = 加入者（B 端）              // ← B 端被当成"AI"对待！
if (state.mode === 'online') {
  startTurn('ai');                       // ← 联机借用人机分支
}
```

**导致**：
1. ❌ 一个回合走完就卡死（B 端没独立的回合处理逻辑）
2. ❌ A 端断线 = 游戏死（state owner 没了）
3. ❌ 无法反作弊（客户端可改 state）
4. ❌ 无法做观战/录像
5. ❌ 5 阵营扩展要改"人机+联机"两份逻辑

### 0.2 V5.0 目标

**重新设计为"A vs B 平等对战"架构**，由独立服务器当裁判：

- ✅ **A 和 B 完全平等**（都用 `A`/`B` 命名，不用 `player`/`ai`）
- ✅ **服务器持 state**（center server，唯一权威）
- ✅ **客户端只发 action 指令**（不出牌/攻击/结束回合）
- ✅ **服务器验证 + 执行 + 广播**
- ✅ **断线重连无感**（服务器保留 state）
- ✅ **5 阵营扩展**改一套（只针对联机对战）

## 一、架构总览

```
┌─────────────────────────────────────────────────────────────┐
│                      Center Server (Node.js)                │
│                                                              │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ Room 1   │  │ Room 2   │  │ Room 3   │  │  ...     │   │
│  │ state:{} │  │ state:{} │  │ state:{} │  │          │   │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │
│                                                              │
│  职责：                                                       │
│  • 持有每个房间的完整 game state                              │
│  • 验证客户端发来的 action 合法性                            │
│  • 执行 action（修改 state）                                 │
│  • 广播新 state 给所有客户端                                 │
│  • 判定胜负                                                  │
│  • 录像回放（action 序列）                                   │
└─────────────────────────────────────────────────────────────┘
        ↑              ↑              ↑              ↑
        │ws            │ws            │ws            │ws
        │              │              │              │
   [Player A]    [Player B]    [Player C]    [Player D]
   (only render) (only render) (spectator)   (spectator)
```

**关键变化**：
- **服务器 = 唯一权威**（state owner）
- **客户端 = 纯渲染器**（无业务逻辑）
- **action 协议**（不是 state 镜像）

## 二、概念重命名（核心）

| 概念 | V4.x | V5.0 |
|------|------|------|
| 当前回合方 | `state.turn = 'player' \| 'ai'` | `state.turn = 'A' \| 'B'` |
| 玩家 A 数据 | `state.player` | `state.A` |
| 玩家 B 数据 | `state.ai` | `state.B` |
| 我的阵营 | `myOnlineSide = 'player' \| 'ai'` | `mySide = 'A' \| 'B'` |
| 玩家 A 指挥点 | `state.k.player` | `state.k.A` |
| 玩家 B 指挥点 | `state.k.ai` | `state.k.B` |
| 玩家 A 的 HQ | `state.player.hq` | `state.A.hq` |
| 玩家 B 的 HQ | `state.ai.hq` | `state.B.hq` |
| 玩家 A 的手牌 | `state.player.hand` | `state.A.hand` |
| 玩家 B 的手牌 | `state.ai.hand` | `state.B.hand` |

**好处**：
- 代码读起来是"两个 player"，不混淆
- 联机模式下不需要特殊分支
- 添加 C、D 玩家（观战/未来多人）零成本

## 三、协议设计（V5.0 vs V4.x）

### 3.1 V4.x 协议（state 镜像）

```
客户端 A → wsSend({ type: 'state', state: <完整 state 对象> })
                ↓
              服务端 → 中继 → B 端
                ↓
        B 端覆盖自己 state → render
```

**问题**：
- state 大（包含双方手牌、支援线、HQ...）
- 客户端可伪造 state
- 带宽浪费

### 3.2 V5.0 协议（action 指令）

```
客户端 A → wsSend({ type: 'action', kind: 'play_card', cardIdx: 2 })
                ↓
              服务端
                ↓
          验证：是不是 A 的回合？
          验证：手牌 idx=2 是不是 A 的？
          验证：K 值够不够？
                ↓
          执行：state.A.hand.splice(2, 1)
                state.A.support[slot] = card
                ↓
          广播：{ type: 'state', state: <新 state> } → A 端 + B 端
                ↓
          A 端收到 state → render
          B 端收到 state → render
```

**优势**：
- 客户端只发"我做了什么"（小消息）
- 服务器是唯一改 state 的人
- 带宽小（action 几十字节 vs state 几 KB）
- 反作弊简单

### 3.3 action 协议类型

```js
// 1. 出牌（部署支援线）
{ type: 'action', kind: 'play_card', cardIdx: 0, slot: 1 }
//   cardIdx: 手牌中的索引
//   slot: 支援线空槽位（0-4）

// 2. 攻击
{ type: 'action', kind: 'attack', fromInstance: 'u_abc', toInstance: 'u_xyz' }
//   fromInstance: 攻击方单位 ID
//   toInstance: 目标单位 ID（或 'hq_B' 表示对方 HQ）

// 3. 推进前线
{ type: 'action', kind: 'advance', fromInstance: 'u_abc', toSlot: 2 }
//   toSlot: 前线空槽位

// 4. 结束回合
{ type: 'action', kind: 'end_turn' }

// 5. 指令牌（op 卡）的特殊效果
{ type: 'action', kind: 'op_card', cardIdx: 0, target: '...可选目标' }

// 6. 接受/拒绝（如投降/求和）
{ type: 'action', kind: 'surrender' }
```

### 3.4 state 协议（服务端 → 客户端）

```js
// 完整 state 广播（每次 action 后）
{ type: 'state', state: <完整 game state> }

// 增量通知（如回合开始、胜负）
{ type: 'event', event: 'turn_start', player: 'A', round: 2 }
{ type: 'event', event: 'game_end', winner: 'B' }
```

## 五、服务器设计（game-engine.js）

### 5.1 核心模块

```js
// game-engine.js（重写，原 game-server.js 替换）

// 1. 房间管理
function createRoom() {}
function joinRoom(ws, roomId, side) {}
function destroyRoom(roomId) {}

// 2. State 初始化
function newGame(room, firstPlayer) {}
function startTurn(room, who) {}
function drawTo(room, who) {}

// 3. Action 验证（核心）
function validateAction(room, side, action) {
  // 1. 是不是当前回合方？
  if (room.state.turn !== side) return { ok: false, err: '不是你的回合' };
  
  // 2. 验证具体 action
  switch (action.kind) {
    case 'play_card':
      return validatePlayCard(room, side, action);
    case 'attack':
      return validateAttack(room, side, action);
    case 'end_turn':
      return { ok: true };
    // ...
  }
}

// 4. Action 执行
function executeAction(room, side, action) {
  switch (action.kind) {
    case 'play_card':
      // 从手牌移到支援线
      const card = room.state[side].hand[action.cardIdx];
      room.state[side].support[action.slot] = makeUnit(card, side);
      room.state[side].hand.splice(action.cardIdx, 1);
      room.state.k[side].cur -= card.cost.deploy;
      break;
    case 'attack':
      doAttack(room, side, action);
      break;
    case 'end_turn':
      endTurn(room, side);
      break;
  }
  
  // 检查胜负
  if (checkVictory(room)) {
    room.state.phase = 'ended';
  }
}

// 5. 主流程（处理 action 消息）
case 'action': {
  const room = rooms.get(myRoomId);
  
  // 验证
  const v = validateAction(room, mySide, msg);
  if (!v.ok) {
    ws.send({ type: 'error', msg: v.err });
    return;
  }
  
  // 执行
  executeAction(room, mySide, msg);
  
  // 广播
  broadcast(room, { type: 'state', state: room.state });
}
```

### 5.2 状态机

```
[init] 
   ↓ newGame
[round 1, turn=firstPlayer]
   ↓ action
[validate + execute]
   ↓
[check victory?]
   ↓
   ├─ yes → [phase=ended, winner=X] → broadcast
   ↓ no
   ├─ end_turn action → startTurn(next) → broadcast
   ↓
[round N, turn=next]
   ↓
   ...循环...
```

## 六、客户端设计（重构 game-state.js）

### 6.1 客户端只做 3 件事

```js
// 1. 发 action
function sendAction(action) {
  wsSend({ type: 'action', ...action });
}

// 2. 收 state 后 render
case 'state':
  state = received;
  render();
  break;

// 3. 收事件后显示提示
case 'event':
  showNotification(received.event, received);
  break;
```

### 6.2 客户端发 action 的位置

| 用户操作 | 客户端发送 |
|---------|-----------|
| 点手牌 → 选空槽 | `sendAction({ kind: 'play_card', cardIdx, slot })` |
| 点我方单位 → 点敌方单位 | `sendAction({ kind: 'attack', from, to })` |
| 点我方支援线单位 → 选前线空槽 | `sendAction({ kind: 'advance', from, toSlot })` |
| 点"结束回合" | `sendAction({ kind: 'end_turn' })` |
| 点指令牌 | `sendAction({ kind: 'op_card', cardIdx, target? })` |

### 6.3 canAct 的简化

```js
// V4.x: 复杂，区分模式
function canAct() {
  if (state.mode === 'ai') return state.turn === 'player';
  if (state.mode === 'online') return state.turn === myOnlineSide;
  // ...
}

// V5.0: 统一，联机/人机都用一个判断
function canAct() {
  return state.turn === mySide;  // 不管模式，永远是"是不是我方回合"
}
```

```

        [round=1, turn=firstPlayer, phase=main]
                        ↓
        ┌───────────────[action]───────────────┐

   ↓

   ↓
```

## 八、关键能力（V5.0 新增）

### 8.1 断线重连

```js
// 客户端断线 5 秒后自动重连
wsConn.onclose = function() {
  setTimeout(() => {
    wsConnect(() => {
      wsSend({ type: 'rejoin', roomId, side });
    });
  }, 5000);
};

// 服务端处理 rejoin
case 'rejoin': {
  // 找到房间
  const room = rooms.get(roomId);
  if (!room) return error;
  
  // 恢复 ws 连接
  room.players[side].ws = ws;
  room.players[side].isOnline = true;
  
  // 发送完整 state（断线期间的游戏进度）
  ws.send({ type: 'state', state: room.state });
  
  // 如果是 A 的回合，A 可能需要重新决策
  // 如果是 B 的回合断线了，A 端可以"代替"操作吗？（V5.0 MVP 先不支持）
}
```

### 8.2 录像回放

```js
// 服务器记录所有 action
room.state.actionLog.push({
  time: Date.now(),
  player: side,
  kind: action.kind,
  data: action,
});

// 玩家看录像：发 replay 请求
{ type: 'replay', roomId, actionLog: [...] }

// 服务端返回 action 序列
{ type: 'replay_data', actions: [...] }
```

### 8.3 观战模式

```js
// 观战者加入房间（只读）
case 'spectate': {
  room.spectators.push(ws);
  // 发送当前 state（不参与游戏）
  ws.send({ type: 'state', state: room.state });
  // 后续每次 state 变化也广播给观战者
}

// 区分 ws 消息路由
broadcast(room, msg) {
  [...playerWs, ...spectatorWs].forEach(ws => ws.send(msg));
}
```

### 8.4 反作弊

服务器验证所有 action：
- ✅ 是不是当前回合方？
- ✅ K 值够不够？
- ✅ 目标是不是合法的？
- ✅ 单位是不是已部署的（不是凭空捏造的）？
- ✅ 支援线槽位是不是空的？

## 十四、附录：完整 action 协议

```typescript
// 出牌（支援线部署）
type PlayCardAction = {
  kind: 'play_card';
  cardIdx: number;     // 手牌索引
  slot: number;        // 支援线槽位 0-4
};

// 攻击
type AttackAction = {
  kind: 'attack';
  fromInstance: string;  // 攻击方单位 instanceId
  toInstance: string;    // 目标 instanceId 或 'hq_A'/'hq_B'
};

// 推进前线
type AdvanceAction = {
  kind: 'advance';
  fromInstance: string;  // 支援线单位
  toSlot: number;        // 前线槽位 0-4
};

// 结束回合
type EndTurnAction = {
  kind: 'end_turn';
};

// 指令牌（op 卡）
type OpCardAction = {
  kind: 'op_card';
  cardIdx: number;
  target?: string;       // 可选目标
};

// 投降
type SurrenderAction = {
  kind: 'surrender';
};

type GameAction = PlayCardAction | AttackAction | AdvanceAction | EndTurnAction | OpCardAction | SurrenderAction;
```
